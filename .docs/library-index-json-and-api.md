# Library index: JSON format and API

This document answers: **Is there a JSON with full library description? Is it updated? Is there an API to get it?**

## Is there a JSON with full library description?

**Yes.** The library is stored as structured JSON in the content store (e.g. `.content` or your nx-content backend).

| Document | Path | Contents |
|----------|------|----------|
| **Aggregate index** | `skills/index.v1.json` | Single file: `schemaVersion`, `generatedAt`, `skills[]` where each item is `{ "$refKey": "skills/index/v1/<skillId>.json" }`. |
| **Per-skill (full description)** | `skills/index/v1/<skillId>.json` | One file per skill with full metadata: `id`, `displayName`, `description`, `source` (files, contentHash), `runtime` (callName, modes, defaults), `io` (input/output JSON Schema), optional `examples`, `tags`, `quality` (confidence, notes). |
| **Meta/report** | `skills/index/v1/_meta.json` | Generation stats and errors from the last index run. |

The **full library description** for the whole catalog is therefore: the aggregate file (list of refs) plus the set of per-skill JSON files. The canonical schema for these is in **`.docs/skills-index.schema.v1.json`**. For a full field-by-field description, see [skills-index.v1.md](skills-index.v1.md).

## Is it updated?

**Yes.** The index is regenerated by:

1. **HTTP API**
   - **`POST /content/index`**  
     Body (optional): `{ "prefix": "skills/" }`.  
     Runs `updateLibraryIndex`, writes `skills/index.v1.json`, per-skill files under `skills/index/v1/`, and `_meta.json`.  
     Response: `{ "indexed", "skills", "errors" }`.

2. **CLI**
   - **`npx tsx scripts/updateLibraryIndex.ts`**  
     Options: `--dry-run`, `--incremental`, `--force`, `--prefix=...`, `--mode=weak|normal|strong`, `--model=...`.  
     Uses the same `updateLibraryIndex()` from `src/content/libraryIndex.js`; requires a `.content` skills subtree and (for normal/strong) `OPENROUTER_API_KEY`.

3. **Content sync**  
   The content-sync flow can also run index updates (see `handleContentSync` / `handleContentIndex` in `src/serve.ts`).

So the JSON is **updated whenever you run the index build** (API or CLI); it is not auto-updated on every code change unless you wire that in (e.g. CI or a post-sync hook).

## Is there an API to get it?

**Partially.** The server does **not** expose the raw content-store files (e.g. you cannot `GET /content/skills/index.v1.json`). It does expose a **derived** view of the library over HTTP:

| Endpoint | Method | What you get |
|----------|--------|----------------|
| **`/functions`** | GET | **List view:** `{ "skills": [ { "name", "description", "category", "tags", "modes", "version", "input", "output" } ] }`. Optional query: `?tag=...`, `?category=...`, `?q=...`. Built from the library index + per-skill entries; not the raw aggregate JSON. |
| **`/functions/:id`** | GET | **Full description for one skill:** the full per-skill index entry (same shape as `skills/index/v1/<skillId>.json`), plus `name`, `version`, `examples`, `instructions`. So you can get the full library description **per skill** by calling this for each skill id. |
| **`/content/index`** | POST | **Update** the index (see above). It does not **return** the full library JSON; it returns `{ "indexed", "skills", "errors" }`. |

So:

- **Full library as a single JSON blob:** not served by the API. You would need to either read the content store directly (e.g. from the repo or your nx-content backend) or call **GET /functions** to get the list of skill ids, then **GET /functions/:id** for each id to assemble the full per-skill descriptions.
- **List of skills with descriptions and I/O:** **GET /functions**.
- **Full description for one skill:** **GET /functions/:id**.

## Summary

| Question | Answer |
|----------|--------|
| JSON with full library description? | Yes: `skills/index.v1.json` (aggregate) + `skills/index/v1/<skillId>.json` (per-skill). Schema: `.docs/skills-index.schema.v1.json`. |
| Is it updated? | Yes, when you run the index build via **POST /content/index** or **`npx tsx scripts/updateLibraryIndex.ts`** (and optionally via content-sync). |
| API to get it? | No single “download full library JSON” endpoint. Use **GET /functions** for a list with descriptions and I/O; use **GET /functions/:id** for the full per-skill description. To get the exact index files, read them from the content store. |

For index schema details, LLM usage, and validation, see [skills-index.v1.md](skills-index.v1.md).
